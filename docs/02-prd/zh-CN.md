## 第零章：范围与验收

### 0.1 Foundation 目标

`v0.1.0-alpha` 的目标是建立一个可重复构建、可启动、可审查的工程起点。该版本不交付 RAG、Agent、Workflow、Connector 或企业治理业务功能。

### 0.2 Foundation 需求

| ID | 优先级 | 需求 | 验收标准 |
|---|---|---|---|
| FND-001 | P0 | 治理基线 | OEP、RFC、ADR、发布、安全和贡献流程有明确入口且无互相冲突的规则 |
| FND-002 | P0 | 产品与架构基线 | PRD、SAD、SDD 标明版本、状态、范围和未验证假设 |
| FND-003 | P0 | Java 脚手架 | JDK 21 环境下 Wrapper 构建、静态检查和测试通过，服务可返回健康状态 |
| FND-004 | P0 | Python 脚手架 | Python 3.12 环境下可安装、Lint、测试并启动 FastAPI 健康端点 |
| FND-005 | P0 | Frontend 脚手架 | Node.js 22 环境下 `npm ci`、Lint、格式检查和生产构建通过 |
| FND-006 | P1 | 文档站 | Docusaurus 可执行生产构建，断链使构建失败 |
| FND-007 | P0 | 容器入口 | 根目录 `docker compose up --build` 可启动 Foundation 服务 |
| FND-008 | P0 | CI | Pull Request 自动执行构建、测试、覆盖率、格式、Compose 和安全扫描 |

### 0.3 非功能要求

| ID | 要求 | Foundation 标准 |
|---|---|---|
| NFR-001 | 可重复性 | 锁定 Node 依赖，使用 Gradle Wrapper，容器基础镜像使用固定版本 |
| NFR-002 | 安全 | 仓库不得包含生产 Secret，HIGH/CRITICAL 扫描结果阻止合并 |
| NFR-003 | 可维护性 | 测试覆盖率不低于 80%，公开契约变更通过 RFC/ADR |
| NFR-004 | 可观测性 | 所有可运行服务至少提供健康端点；业务指标在对应模块实现 |
| NFR-005 | 兼容性 | `0.x` 同一 MINOR 的 PATCH 保持兼容，跨 MINOR 的破坏性变更提供迁移说明 |

### 0.4 后续版本非范围

路线图中的功能描述是产品假设，不代表当前实现或发布承诺。每个后续模块必须补充用户故事、验收标准、子 SDD、威胁模型和基准指标后才能进入开发。

## 第一章：项目背景

### 1.1 为什么企业需要 AI

人工智能正在从实验性工具进入企业业务流程。OpenEIP 聚焦于落地过程中反复出现的工程问题：

- **技术门槛高**：LLM、RAG、Agent、向量数据库等技术栈复杂
- **数据孤岛严重**：ERP、CRM、文档系统、数据库各自独立，AI 无法统一访问
- **缺乏企业级治理**：权限、审计、模型管理、成本控制在大多数 AI 工具中是空白
- **集成成本巨大**：每个系统单独对接 AI 能力，重复建设

### 1.2 企业痛点全景

| 痛点 | 现状 | 后果 |
|---|---|---|
| 数据孤岛 | 数据分散在 10+ 系统中 | AI 只能看到碎片信息 |
| 文档无法搜索 | 80% 企业文档是非结构化的 | 知识无法被检索和复用 |
| Excel 泛滥 | 业务人员依赖 Excel 做分析 | 数据不一致，无法自动化 |
| 审批复杂 | 多系统、多角色、多级审批 | 流程低效，缺乏智能路由 |
| 系统太多 | 平均企业使用 100+ SaaS 工具 | 信息割裂，无统一入口 |
| 没有统一权限 | 每个系统独立管理权限 | 安全风险，管理成本高 |
| 知识流失 | 员工离职带走隐性知识 | 组织记忆消失 |
| AI 不能真正执行任务 | 大多数 AI 产品只能"聊天" | 无法端到端自动化 |

### 1.3 为什么不只使用通用 AI 助手

通用 AI 助手及其企业版本可以解决对话、检索和部分集成需求，但 OpenEIP 关注的是可自托管、跨系统且可扩展的平台层问题：

- 对异构本地系统的统一扩展契约
- 与企业既有权限模型和数据权限的映射
- 可审计的长流程执行与人工审批
- 模型和部署供应商的可替换性

### 1.4 为什么不只使用 AI 应用编排平台

Dify 等项目已经提供成熟的 AI 应用、Workflow、插件和知识能力。OpenEIP 不以功能清单否定这些项目，而是验证另一组设计重点：

- Java 领域事务与 Python AI 执行的明确边界
- 企业数据权限、审计和元数据治理的统一契约
- Connector、Agent 和 Workflow Node 的长期兼容策略
- AI 对话之外的审批、分析和系统执行场景

### 1.5 为什么不是 DataHub

DataHub 是优秀的数据治理平台，但它：

- 聚焦于元数据管理和数据发现
- 不具备 AI 编排能力（Agent/Workflow）
- 不具备 LLM 和 RAG 能力
- 是数据治理工具，不是 AI 操作系统

### 1.6 为什么需要 Enterprise AI Platform

市场需要一个**统一**的平台，同时具备：

- **AI 能力**：RAG、Agent、Workflow（类似 Dify + LangGraph）
- **数据治理**：元数据管理、数据血缘（类似 DataHub + OpenMetadata）
- **连接器生态**：30+ 企业系统连接器（类似 Airbyte）
- **企业级平台**：多租户、RBAC、审计、合规（类似 Spring Boot 企业应用）
- **BI 与分析**：自然语言查询、自动报表（类似 ThoughtSpot）

## 第二章：产品定位

### 2.1 一句话介绍

**OpenEIP 是面向企业的 AI 操作系统——连接、理解、分析、执行、治理企业全部数据与流程。**

### 2.2 目标用户

| 角色 | 需求 | 使用场景 |
|---|---|---|
| **企业管理层** | 数据驱动决策 | Dashboard、BI 报告、异常预警 |
| **业务人员** | 自然语言查询数据 | 用中文查数据库、自动生成报表 |
| **数据分析师** | 统一数据访问 | 跨系统数据分析、知识库检索 |
| **IT 管理员** | 企业治理 | 权限管理、审计、模型管理 |
| **开发者** | 插件开发 | 开发 Connector/Agent/Workflow Node |
| **AI 工程师** | Agent 编排 | 构建 Multi-Agent、Workflow 自动化 |

### 2.3 核心价值

| 价值 | 说明 |
|---|---|
| **统一入口** | 一个平台访问企业全部数据 |
| **AI 原生** | 所有操作都可以通过自然语言完成 |
| **企业就绪** | 权限、审计、合规开箱即用 |
| **开放生态** | Connector Marketplace + Agent Marketplace |
| **端到端自动化** | 从数据连接到 AI 执行全链路 |

### 2.4 产品理念

#### AI Native

AI 不是外挂功能，而是平台的核心引擎。每个模块都原生集成 AI 能力：搜索是 AI 搜索、分析是 AI 分析、审批是 AI 辅助决策。

#### Data Native

数据不是事后导入，而是实时连接。Connector 生态确保平台可以即时访问企业所有数据源，无需数据迁移。

#### Workflow Native

流程不是静态配置，而是 AI 驱动的动态编排。Agent 可以理解上下文、做出决策、执行多步骤任务。

#### Knowledge Native

知识不是简单的文档存储，而是可被 AI 理解、检索、推理的结构化知识图谱。

## 第三章：企业痛点详解

### 3.1 数据孤岛

**问题**：企业数据分散在 ERP（SAP/Oracle）、CRM（Salesforce）、数据库（MySQL/PostgreSQL）、文件系统（NAS/SharePoint）、SaaS 工具（飞书/企业微信/Jira/Confluence）等数十个系统中。

**后果**：
- 查询一个客户 360 视图需要登录 5+ 系统
- 跨系统数据分析几乎不可能
- AI 只能看到部分数据，回答不完整

### 3.2 文档无法搜索

**问题**：企业 80% 的数据是非结构化文档（PDF、Word、Excel、PPT、扫描件、邮件）。

**后果**：
- 员工平均每周花 3.5 小时搜索信息
- 大量知识隐藏在文档中，无法被检索
- 新人入职学习成本极高

### 3.3 Excel 太多

**问题**：业务人员依赖 Excel 做数据整理、分析、报表，每个部门维护自己的 Excel。

**后果**：
- 数据不一致（"哪个版本是最新的？"）
- 无法自动化更新
- 人工错误风险高

### 3.4 审批复杂

**问题**：企业审批涉及多系统、多角色、多级审批，流程僵化。

**后果**：
- 一个合同审批可能需要 2 周
- 审批规则硬编码，无法智能路由
- 缺乏 AI 辅助决策

### 3.5 系统太多

**问题**：平均企业使用 100+ SaaS 工具，每个工具有独立的账号、权限、界面。

**后果**：
- 员工每天在多个系统间切换
- 信息碎片化，缺乏统一视图
- 培训成本高

### 3.6 没有统一权限

**问题**：每个系统独立管理权限，无法实现企业级统一权限管控。

**后果**：
- 离职员工可能保留某些系统的访问权限
- 权限审计需要逐个系统检查
- 无法实现跨系统的数据权限控制

### 3.7 知识流失

**问题**：员工离职带走隐性知识——那些没有写在文档里的经验、判断、技巧。

**后果**：
- 关键岗位离职导致业务中断
- 新人无法快速上手
- 组织知识无法积累

### 3.8 AI 不能真正执行任务

**问题**：大多数企业 AI 工具只能"聊天"，不能真正端到端执行任务。

**后果**：
- AI 可以告诉你"怎么做"，但不能替你做
- 无法触发实际业务流程（如创建工单、发起审批、更新数据库）
- 人工仍然是流程的瓶颈

## 第四章：解决方案

### 4.1 连接（Connect）

通过 **Connector Framework** 统一连接企业全部数据源：

- 数据库：MySQL、PostgreSQL、Oracle、SQL Server、ClickHouse
- 文件系统：NAS、S3、MinIO、SharePoint
- SaaS 工具：飞书、企业微信、Jira、Confluence、GitHub、GitLab
- 企业应用：SAP、Salesforce、用友、金蝶
- 消息系统：Kafka、RabbitMQ

### 4.2 理解（Understand）

通过 **AI Pipeline** 将原始数据转化为可理解的知识：

- **文档解析**：OCR、PDF、Word、Excel、PPT 自动解析
- **Embedding**：文本向量化，支持语义搜索
- **实体识别**：自动识别文档中的人名、公司、日期、金额等实体
- **知识图谱**：构建企业知识图谱，关联实体和关系
- **自动分类与标签**：AI 自动对文档分类和打标签

### 4.3 分析（Analyze）

通过 **AI 分析引擎** 提供自然语言数据查询和分析：

- **SQL Agent**：用中文描述需求，自动生成 SQL 并执行
- **BI Agent**：自动生成图表、Dashboard、数据报告
- **异常检测**：AI 自动识别数据异常并预警
- **预测分析**：基于历史数据进行趋势预测
- **多维分析**：跨数据源的关联分析

### 4.4 执行（Execute）

通过 **Agent + Workflow Engine** 实现端到端任务自动化：

- **Workflow Engine**：可视化编排工作流，支持审批、条件、循环
- **Multi-Agent**：多个 AI Agent 协作完成复杂任务
- **Tool Calling**：Agent 可调用 API、数据库、文件系统等工具
- **MCP 协议**：标准化的 AI 工具调用协议
- **自动触发**：事件驱动的自动任务执行

### 4.5 治理（Govern）

通过 **企业治理框架** 确保安全、合规、可控：

- **多租户**：租户隔离，数据安全
- **RBAC/ABAC**：基于角色和属性的权限控制
- **审计日志**：所有操作可追溯
- **模型管理**：模型版本、评估、切换
- **Prompt 管理**：Prompt 版本、测试、发布
- **成本控制**：Token 用量统计、预算预警
- **安全防护**：Prompt Injection 防护、数据脱敏、加密

## 第五章：产品模块

### 5.1 Connector（连接器）

**定位**：连接企业数据源的插件化框架。

**核心能力**：
- 统一的 Connector SPI：所有连接器遵循统一接口
- 连接管理：创建、测试、监控、删除连接
- 数据同步：全量/增量同步、定时调度
- 元数据提取：自动提取表结构、字段、关系
- 权限映射：数据源的权限映射到平台权限

**首批连接器**：
MySQL、PostgreSQL、Oracle、SQL Server、Redis、Kafka、MinIO、飞书、企业微信、Jira、Confluence、GitHub、GitLab、S3

### 5.2 Knowledge（知识库）

**定位**：企业知识的存储、检索和管理中心。

**核心能力**：
- 文档管理：上传、解析、分类、版本管理
- AI 解析：OCR、PDF、Word、Excel、PPT、图片
- 向量化：Embedding 生成与存储（Milvus）
- 全文检索：Elasticsearch 集成
- 语义搜索：RAG（检索增强生成）
- 知识图谱：Neo4j 实体关系存储
- 引用溯源：AI 回答附带来源引用

### 5.3 Agent（智能体）

**定位**：AI 驱动的自主任务执行单元。

**核心能力**：
- Agent Runtime：Agent 生命周期管理
- Tool Calling：标准化的工具调用机制
- Memory：短期记忆（对话上下文）+ 长期记忆（向量存储）
- Planner：任务规划和分解
- Executor：逐步执行和结果验证
- Reflection：自我反思和纠错
- Multi-Agent：多 Agent 编排和协作
- MCP 支持：Model Context Protocol 标准接入

**首批 Agent 类型**：
Document Agent、SQL Agent、BI Agent、Search Agent、Workflow Agent

### 5.4 Workflow（工作流）

**定位**：可视化的工作流编排和执行引擎。

**核心能力**：
- Canvas 编辑器：拖拽式工作流设计
- 节点类型：LLM / Agent / Tool / Condition / Loop / Approval / Delay / Webhook
- 执行引擎：并行/串行执行、错误重试、超时控制
- 触发器：定时、Webhook、事件驱动（Kafka）
- 审批集成：多级审批、会签、或签、转审
- 版本管理：工作流版本发布和回滚

### 5.5 BI（商业智能）

**定位**：AI 驱动的自然语言数据分析。

**核心能力**：
- SQL Agent：自然语言转 SQL
- 自动图表：根据数据自动选择最佳可视化
- Dashboard：可拖拽的仪表盘设计器
- 报表生成：自动生成分析报告（PDF/Word）
- 异常检测：时序数据异常自动发现
- 预测分析：基于历史数据的趋势预测

### 5.6 Search（企业搜索）

**定位**：跨数据源的统一搜索引擎。

**核心能力**：
- 跨源搜索：同时搜索文档、数据库、SaaS 工具
- 语义搜索：理解用户意图，不依赖关键词
- 权限过滤：搜索结果自动应用数据权限
- 个性化排序：基于角色和历史的智能排序
- 搜索建议：自动补全和相关推荐

### 5.7 Governance（治理）

**定位**：企业级安全、合规和管理中心。

**核心能力**：
- 多租户管理：租户创建、配额、隔离
- 权限管理：RBAC + ABAC，数据行级权限
- 审计日志：操作审计、数据访问审计、AI 调用审计
- 模型治理：模型版本管理、A/B 测试、评估
- Prompt 治理：Prompt 模板、版本、测试
- 成本治理：Token 用量、预算、预警
- 安全策略：Prompt Injection 防护、数据脱敏、IP 白名单

### 5.8 Plugin（插件生态）

**定位**：开放平台的核心——让社区贡献扩展能力。

**核心能力**：
- Connector SDK：开发新的数据源连接器
- Agent SDK：开发新的 Agent 类型
- Plugin SDK：开发通用插件
- Marketplace：插件发布、发现、安装
- 版本管理：插件版本兼容性检查

## 第六章：竞争分析

### 6.1 竞争格局

| 产品类别 | 参考项目 | OpenEIP 需要验证的问题 |
|---|---|---|
| AI 应用与 Agent 编排 | Dify、Flowise、Langflow | 企业权限、长流程可靠性和扩展契约是否形成真实差异 |
| 企业知识与对话 | OpenWebUI 等 | 跨数据源权限和可追溯检索是否有足够需求 |
| 元数据与治理 | DataHub、OpenMetadata | AI 执行与元数据治理结合是否优于松耦合集成 |
| 数据集成 | Airbyte 等 | Connector SDK 是否能在实时访问、同步和安全之间取得平衡 |
| 企业协作 AI | Microsoft、飞书、企业微信生态 | 开放部署和跨生态能力是否能抵消原生套件集成优势 |

### 6.2 OpenEIP 的差异化优势

竞争能力必须以可复现版本为单位评估。功能存在性、许可证、部署模式、权限粒度、扩展兼容性和运行成本都要保留证据链接；不得用简单的是/否矩阵替代调研。

**差异化假设**：OpenEIP 希望通过统一的扩展契约和企业治理基线组合 AI 编排、数据接入与业务流程能力。该假设需要通过用户访谈、可运行版本和公开 Benchmark 持续验证，不作“市场唯一”声明。

### 7.2 社区建设

- GitHub Discussions 作为社区交流主阵地
- RFC 流程确保社区参与重大决策
- Good First Issue 降低贡献门槛
- 定期发布 Release Notes 和 Blog

### 7.3 商业版规划（未来）

- 核心平台、SSO/LDAP、标准高可用部署能力保持开源路线
- 商业化范围限于支持服务、托管服务、实施咨询和经过授权的第三方插件
- 社区版与商业服务的边界必须通过公开 RFC 决定

## 附录

### A. 术语定义

| 术语 | 定义 |
|---|---|
| Connector | 数据源连接插件，遵循统一 SPI |
| Agent | AI 驱动的自主任务执行单元 |
| Workflow | 可视化编排的多步骤业务流程 |
| RAG | 检索增强生成：检索 + LLM 生成 |
| MCP | Model Context Protocol：AI 工具调用标准协议 |
| SPI | Service Provider Interface：插件扩展接口 |

### B. 参考项目

- Spring Boot / Spring Cloud：企业工程规范参考
- Dify：AI 编排参考
- DataHub / OpenMetadata：数据治理参考
- Airbyte：连接器生态参考
- Temporal / Kestra：工作流引擎参考
- LangGraph / PydanticAI：Agent 框架参考

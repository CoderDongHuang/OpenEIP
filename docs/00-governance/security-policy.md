# Security Policy（安全策略）

> 本文档定义 OpenEIP 的安全漏洞报告流程、响应 SLA 和披露政策。

---

## 安全漏洞报告

### 报告渠道

如果你发现安全漏洞，**请不要在公开 Issue 中报告**。请通过以下渠道私密报告：

- **GitHub Security Advisory**：通过 GitHub 的 "Report a vulnerability" 功能

### 报告内容

请尽可能包含以下信息：

- 漏洞描述
- 复现步骤
- 影响范围（哪些版本受影响）
- 可能的修复建议
- 你的联系方式

## 响应 SLA

| 严重程度 | 初始响应 | 修复时间 | 发布方式 |
|---|---|---|---|
| **CRITICAL** | 24 小时内 | 72 小时内 | 紧急补丁版本 |
| **HIGH** | 48 小时内 | 7 天内 | 下个 PATCH 版本 |
| **MEDIUM** | 1 周内 | 30 天内 | 下个 MINOR 版本 |
| **LOW** | 2 周内 | 90 天内 | 下个 MINOR 版本 |

### 严重程度定义

| 级别 | 定义 |
|---|---|
| **CRITICAL** | 无需认证的远程代码执行、任意数据泄露 |
| **HIGH** | 认证绕过、权限提升、敏感信息泄露 |
| **MEDIUM** | 特定条件下的安全问题、需用户交互的攻击 |
| **LOW** | 理论上的安全问题，实际利用难度高 |

## 披露政策

### 协调披露（Coordinated Disclosure）

1. 报告者私密报告漏洞
2. 维护者确认并修复
3. 修复版本发布
4. 发布 Security Advisory
5. 在 CVE 数据库记录（如适用）
6. 在 CHANGELOG 中记录

### 漏洞奖励

目前 OpenEIP 尚未提供漏洞奖励计划。我们将在适当的时机建立。

## 安全最佳实践

### 开发阶段

- 所有 PR 通过 Security Review（Quality Gate 第 4 项）
- 使用 Dependabot / Renovate 自动更新依赖
- CI 中集成 SAST 工具

### 运行时

- 使用最小权限原则
- Secret 不硬编码，使用环境变量或 Secret 管理服务
- 启用审计日志
- 定期进行安全扫描
